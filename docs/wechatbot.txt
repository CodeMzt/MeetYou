MeetYou 项目 WeChat iLink Bot 接入方案
目标与背景

MeetYou 是一个以私有服务器为核心的个人智能体系统，支持桌面客户端、飞书 Bot 等多种交互入口。为满足微信用户使用需求，需新增基于 WeChat iLink Bot 协议的接入。本方案依据微信 iLink 官方协议规范，结合 MeetYou 现有 Feishu Bot 架构，设计一个稳定的微信机器人客户端，使用户可通过微信对话与智能体互动，发起任务并接收结果。

方案概述
客户端定位 – WeChat Bot 作为 MeetYou 客户端的扩展实现，与桌面客户端和飞书 Bot 相同，只负责收发消息，不维护全局状态或业务逻辑。
登录认证 – 微信 iLink Bot 采用二维码扫码方式获取授权。首先调用 GET /ilink/bot/get_bot_qrcode 获取二维码及轮询令牌，然后不断调用 GET /ilink/bot/get_qrcode_status 轮询扫码状态；当返回状态为 confirmed 时，将获得 bot_token、ilink_bot_id、ilink_user_id 等凭证。凭证需持久化存储并用于后续业务请求。
消息接收 – 微信 iLink Bot 不支持 WebSocket，而使用 POST /ilink/bot/getupdates 长轮询接口接收消息。客户端需要保持轮询，在服务端返回消息或超时后立即发起下一次请求。返回的 get_updates_buf 是一个不透明游标，用于保持会话位置，应原样保存并在下一次请求中传回。
消息结构与上下文令牌 – 每条入站消息包含 WeixinMessage 结构，字段包括发送者 ID、会话 ID、消息内容列表等。其中最关键的是 context_token 字段，它标识一次会话上下文，发送回复时必须原样回传。
消息发送 – 使用 POST /ilink/bot/sendmessage 接口发送文本或媒体消息。请求体 msg 包含 to_user_id、client_id（本地生成的唯一 ID）、message_type=2（Bot 消息）、message_state=2（完成状态）、context_token 和消息内容数组。发送时需在请求头中带 AuthorizationType: ilink_bot_token、Authorization: Bearer {bot_token} 及随机生成的 X-WECHAT-UIN。
回复流程 – 将微信收到的文本/媒体消息转换成 MeetYou 的统一事件，通过 GatewayConversationClient 传递给 Core。Core 生成回复后，通过 iLink sendmessage 接口回传给用户。同时需要缓存 (from_user_id, context_token) 映射，以便下次收到同一用户消息时可以正确使用最新的上下文令牌。
架构设计
模块组成
模块	说明
WeChatSessionManager	处理二维码登录和凭证管理。封装 get_bot_qrcode、get_qrcode_status 调用及凭证持久化。当会话过期（ret=-14）或用户手动登出时，重新发起扫码流程。
WeChatInputAdapter	类似于 FeishuInputAdapter，负责拉取消息并转换为 MeetYou 事件。包含长轮询协程、游标管理和 context_token 缓存。
WeChatLongPoller	封装 getupdates 长轮询逻辑。处理 get_updates_buf 游标、超时重试和错误码处理。
MessageDispatcher	将微信 WeixinMessage 转换为 MeetYou 协议对象（文本、图片、语音、文件等），并将不同消息类型映射到统一事件结构。
WeChatOutputService	对 sendmessage 调用做封装，生成 client_id，处理文本分片和长消息，组合请求体并发送。
GatewayConversationClient	MeetYou 已有组件。负责向 Core 发送消息并接收回复，用于 QQ / 飞书 / 微信等多端复用。
时序流程

以下流程描述标准的个人私聊场景，群聊暂未公开接口，可依据未来更新扩展。

二维码获取与扫码：
WeChatSessionManager 调用 GET /ilink/bot/get_bot_qrcode?bot_type=3 获取二维码票据和图片地址。前端将二维码呈现给管理员扫码。
后端每隔 3–5 秒调用 GET /ilink/bot/get_qrcode_status?qrcode={token} 轮询状态；根据 status 字段判断下一步：wait 继续等待、scaned 等待手机确认、confirmed 获取 bot_token 等凭证并进入业务阶段、expired 则重新获取二维码。
当返回 confirmed 时保存 bot_token、ilink_bot_id、ilink_user_id 和 baseurl。凭证文件权限需限制为 0600。
建立长轮询：
WeChatLongPoller 启动后立即发送 getupdates 请求，get_updates_buf 为空字符串表示首次拉取。
服务端挂起请求直到有新消息或达到默认超时（约 35 秒）；返回后立即提取 msgs 数组、更新本地 get_updates_buf（持久化）并启动下一轮请求。
若返回 longpolling_timeout_ms 则使用该值作为下一次请求的超时配置。
若返回错误码 -14（session timeout），则停止当前轮询、清除凭证和游标并重启扫码登录。
解析消息并转发：
对返回的每条 WeixinMessage，检查 from_user_id 和 item_list。
过滤来自机器人自身 (from_user_id 以 @im.bot 结尾)；忽略非文本/多媒体内容暂不支持的消息。
将文本内容（MessageItem.type=1、text_item.text）映射为 MeetYou 统一 InboundEvent，并附带 context_token、message_id、session_id 等元数据。
调用 GatewayConversationClient.send_message(text, metadata) 将用户指令发送给 Core。
缓存 (from_user_id, context_token)，更新当次会话的上下文令牌。
处理 Core 回应：
Core 处理完指令后通过 Gateway 返回回复。WeChatOutputService 负责将文本回复打包成 sendmessage 请求：构建 msg 对象并设置 to_user_id 为用户 ID、client_id 为唯一字符串、message_type=2、message_state=2 (FINISH)，context_token 为缓存值。
若文本超过约 2000 字符限制，按协议建议分片，并对每个片段分别调用 sendmessage；每个分片应使用新的 client_id，复用同一 context_token。
请求头需添加 AuthorizationType: ilink_bot_token、Authorization: Bearer {bot_token}、随机 X-WECHAT-UIN 等。

class WeChatInputAdapter:
    def __init__(self, session_manager: WeChatSessionManager, gateway_token: str, event_bus):
        self.session_manager = session_manager
        self.gateway_token = gateway_token
        self.event_bus = event_bus
        self.context_map = {}  # (user_id) -> context_token
        self.get_updates_buf = ""

    async def start(self):
        while True:
            creds = await self.session_manager.ensure_credentials()
            # Pull messages via long poll
            response = await http_post(
                url=f"{creds.baseurl}/ilink/bot/getupdates",
                headers={
                    "Content-Type": "application/json",
                    "AuthorizationType": "ilink_bot_token",
                    "Authorization": f"Bearer {creds.token}",
                    "X-WECHAT-UIN": random_uin(),
                },
                json={"get_updates_buf": self.get_updates_buf, "base_info": {"channel_version": "1.0.0"}},
                timeout=creds.longpoll_timeout,
            )
            if response.ret == -14:
                # Session expired; restart login
                await self.session_manager.invalidate()
                self.get_updates_buf = ""
                continue
            # Update cursor and handle messages
            self.get_updates_buf = response.get("get_updates_buf", "")
            for msg in response.get("msgs", []):
                if msg.get("from_user_id", "").endswith("@im.bot"):
                    continue  # ignore self messages
                context_token = msg.get("context_token")
                from_user = msg.get("from_user_id")
                items = msg.get("item_list", [])
                # Only handle first text item for simplicity
                for item in items:
                    if item.get("type") == 1:  # TEXT
                        text = item.get("text_item", {}).get("text", "").strip()
                        self.context_map[from_user] = context_token
                        client = await self._get_gateway_client(from_user)
                        await client.send_message(text, metadata={
                            "context_token": context_token,
                            "message_id": msg.get("message_id"),
                        })
                        break

    async def send_reply(self, to_user_id: str, text: str):
        creds = await self.session_manager.ensure_credentials()
        context_token = self.context_map.get(to_user_id)
        if not context_token:
            # If missing context_token, cannot reply; log error or request user to initiate chat again
            return
        # Split long texts into fragments of at most 2000 characters
        for fragment in split_text(text, limit=2000):

                "msg": {
                    "to_user_id": to_user_id,
                    "client_id": generate_unique_client_id(),
                    "message_type": 2,
                    "message_state": 2,
                    "context_token": context_token,
                    "item_list": [{"type": 1, "text_item": {"text": fragment}}],
                },
                "base_info": {"channel_version": "1.0.0"},
            }
            await http_post(
                url=f"{creds.baseurl}/ilink/bot/sendmessage",
                headers={
                    "Content-Type": "application/json",
                    "AuthorizationType": "ilink_bot_token",
                    "Authorization": f"Bearer {creds.token}",
                    "X-WECHAT-UIN": random_uin(),

主流方案与参考库

微信 iLink 协议仍处在快速演进阶段，社区已出现若干开源 SDK，可供集成：

weixin-bot (Node.js / Python) – epiral 维护的 SDK，支持二维码登录、长轮询拉取消息、发送文本/媒体和上传文件。文档和示例详尽，易于快速接入。
wechat-robot-go – Go 语言实现，封装扫码登录、长轮询、context_token 缓存和文本/富媒体发送，适合性能要求较高的场景。
openclaw-weixin / wechatbot – Rust / JavaScript 实现，作为 iLink 兼容网关，提供可扩展的插件系统。

选用官方或成熟 SDK 可以避免协议细节差异（如 X-WECHAT-UIN 生成和 client_id 格式）导致的兼容性问题，缩短集成时间。

风险与注意事项
Session 过期 – 服务端在长轮询返回 ret = -14 时表示会话过期。应立即停止请求、清除凭证和游标并重新扫码登录。
长轮询效率 – WeChat Bot 使用 HTTP 长轮询而非 WebSocket，耗用更多连接资源。应为每个 bot 账号单独维护轮询协程并使用合理的超时值（约 35 秒）。
上下文令牌依赖 – 回复时必须回传 context_token，否则无法正确送达消息。应缓存并更新用户的最新 context_token。主动消息在无上下文时通常无法发送。
消息长度限制 – 社区实践建议文本消息不超过约 2000 字符，需自行分片。生成分片时优先在换行或空格处分割。
用户数量与权限 – 当前协议主要支持个人号（私域）Bot，每个 bot 账号只能对特定用户列表（扫码授权用户）发送消息；尚不支持群聊、公众号等场景。集成前需确认业务需求与权限匹配。
安全与合规 – 业务请求需要随机 X-WECHAT-UIN 并确保传输 HTTPS。凭证文件需设置严格权限，防止泄露。
结论

通过新增 WeChatSessionManager、WeChatInputAdapter、WeChatLongPoller 和 WeChatOutputService，可以在 MeetYou 平台接入微信 iLink Bot。该方案基于官方协议规范，采用二维码扫码认证、长轮询消息接收和 context_token 会话路由。结合 MeetYou 现有的 Gateway 和 Feishu 模块，可实现统一的会话管理、消息分发和权限控制，为微信用户提供与其他客户端一致的体验。
