# Workspace / Memory / ContextPool Design

## Purpose

This document records the V3 model for workspace, memory, and cross-client context.

## Workspace Semantics

- A workspace is a logical and sometimes physical affinity scope, not a hard visibility or security boundary.
- Principal/auth remains the real permission boundary.
- Threads keep a home workspace.
- Sessions keep an active workspace.
- Messages record the active workspace at creation time.
- Clients can belong to multiple workspaces.
- Workspace-first behavior means ranking and defaults prefer the active/home workspace, while cross-workspace information can still be retrieved when useful.

## Memory Semantics

- Memory is principal-level and consistent across Clients.
- Runtime memory record ownership uses the principal key/id.
- Client and source ids are attribution metadata, not the default visibility filter.
- Workspace, thread, and session are ranking/tag signals, not independent memory stores.
- Current workspace receives priority, but records from other workspaces are not hidden by default.
- Memory views may still filter by source attribution for operator/UI inspection.
- Stable facts, preferences, and user profile information belong in memory.
- Memory write/delete operations remain bounded by confirmation and risk rules.

## ContextPool Semantics

- ContextPool stores short and medium-term context generated by Clients, tools, and assistant messages.
- It complements memory; it does not replace long-term memory.
- Context building uses current short-tail history, ContextPool recall, and memory recall together.
- ContextPool retrieval is principal-level with boosts for same session, same thread, and same workspace.
- ContextPool keeps a bounded active window per principal.
- User/assistant turns and salient tool results can contribute ContextPool items.
- ContextPool source attribution uses `source_client_id` when a Client row is known.

## Public Interface Direction

- `RuntimeThreadResponse.home_workspace_id` is the canonical thread affinity field.
- `RuntimeSessionResponse.active_workspace_id` is the canonical current workspace field.
- Legacy `workspace_id` remains only as a compatibility alias where needed.
- `PATCH /client/sessions/{session_id}/active-workspace` switches the active workspace.
- `GET /client/context-pool/query` explains currently recalled context pool items.
- `GET /client/workspaces/{workspace_id}/clients?include_tools=true` exposes workspace Client/tool targets.
- Assistant-facing `list_workspaces` and `switch_workspace` use the same active workspace semantics as the HTTP API.

## Acceptance Boundary

- Cross-client continuation should work when two Clients share the same principal, even if their active workspaces differ.
- Current workspace content should rank higher, but unrelated workspace content must not be hard-filtered solely because a workspace is active.
- Local files, Shell, desktop, and edge capabilities execute through directed Client tools, not Core host execution.
- Any schema/API change in this area must include focused backend verification and frontend contract verification.
