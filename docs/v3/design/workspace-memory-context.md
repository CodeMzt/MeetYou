# Workspace / Memory / ContextPool Design

## Purpose

This document records the simplified V3 model for workspace, memory, and cross-client context. It is the active design source for this area.

## Workspace Semantics

- A workspace is a logical and sometimes physical affinity scope, not a hard visibility or security boundary.
- Principal/auth remains the real permission boundary.
- Threads keep a home workspace. This is the default affinity for a conversation.
- Sessions keep an active workspace. This can change while a thread continues.
- Messages record the active workspace at the time they were created.
- Clients and agents can both belong to multiple workspaces.
- Workspace-first behavior means ranking and defaults prefer the active/home workspace, while cross-workspace information can still be retrieved when useful.

## Memory Semantics

- Memory is principal-level and consistent across all clients and agents.
- Workspace, thread, and session are ranking/tag signals, not independent memory stores.
- Current workspace receives priority, but records from other workspaces are not hidden by default.
- Stable facts, preferences, and user profile information belong in memory.
- Memory write/delete operations remain bounded by existing confirmation and risk rules.

## ContextPool Semantics

- ContextPool stores short and medium-term context generated by clients, agents, tools, and assistant messages.
- It complements memory; it does not replace long-term memory.
- Context building uses current short-tail history, ContextPool recall, and memory recall together.
- ContextPool retrieval is principal-level with boosts for same session, same thread, and same workspace.
- Old session summaries and conversation summaries should not become a second long-term context source; new consolidation should go through ContextPool or memory.

## Public Interface Direction

- `ClientThreadResponse.home_workspace_id` is the canonical thread affinity field.
- `ClientSessionResponse.active_workspace_id` is the canonical current workspace field.
- Legacy `workspace_id` remains only as a compatibility alias where needed.
- `PATCH /client/sessions/{session_id}/active-workspace` switches the active workspace.
- `GET /client/context-pool/query` explains which context pool items are currently recalled.
- `GET /client/workspaces/{workspace_id}/clients` exposes client membership for a workspace.
- The assistant-facing `switch_workspace` tool uses the same active workspace semantics as the HTTP API.

## Acceptance Boundary

- Cross-client continuation should work when two clients share the same principal, even if their active workspaces differ.
- Current workspace content should rank higher, but unrelated workspace content must not be hard-filtered solely because a workspace is active.
- Local files, shell, and desktop capabilities still execute through the Agent boundary, not Core.
- Any schema/API change in this area must include at least a focused backend verification path and frontend contract verification.
